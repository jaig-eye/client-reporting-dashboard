1. Project Overview

This is a private, internal web application built and operated exclusively by our marketing agency. Its purpose is to provide a clean, real-time campaign performance dashboard to clients whose Google Ads accounts we manage under our MCC (Manager Account).

The application is not available to the general public. Access is restricted to:

  • Agency administrators — internal staff who manage client accounts and trigger data syncs via a password-protected admin panel.
  • Agency clients — each client receives a unique, single-use token link that shows only their own campaign data. They cannot access any other client's data.

All Google Ads API calls are made server-side (never from a browser). Data is stored in our own database and served to the dashboard from there — the API is not called at dashboard view time.

2. How We Use the Google Ads API

We use the API for one purpose: reading campaign performance metrics for sub-accounts under our MCC. We do not create, modify, or delete any campaigns, budgets, or ads.

Endpoint / Resource Purpose Access Type
customers:listAccessibleCustomers Discover which sub-account IDs are accessible under the MCC after OAuth Read
customers/{id}/googleAds:search GAQL query to fetch daily campaign metrics (spend, clicks, impressions, conversions, ROAS) Read
OAuth 2.0 token refresh Maintain access without requiring the agency to re-authenticate manually Auth

3. Data Accessed

We query the following fields via GAQL — all are aggregated campaign-level metrics with no personally identifiable information:

  • campaign.id and campaign.name
  • segments.date — daily breakdown
  • metrics.cost_micros — ad spend
  • metrics.impressions
  • metrics.clicks
  • metrics.conversions and metrics.conversions_value
  • metrics.ctr

We do not access:

  • User lists, audience data, or any personally identifiable information
  • Billing or payment information
  • Search terms, keywords, or ad creative content
  • Any data outside of our managed MCC sub-accounts

4. Authentication & Token Handling

Authentication is handled via a standard OAuth 2.0 flow using our registered Google Cloud project. The agency owner authenticates once per sub-account. Tokens are stored securely server-side in our database (Supabase) and are never exposed to end users or browsers.

  • Access tokens are refreshed automatically using stored refresh tokens before each sync.
  • The login-customer-id header is set to our MCC ID on every API call, as required.
  • Token storage is encrypted at rest via our database provider.

All Google Ads API requests originate from our server infrastructure. No API credentials or tokens are ever sent to or accessible by end-user browsers.

6. Request Volume & Usage Pattern

API usage is minimal and predictable:

  • One sync per client per day (scheduled nightly cron)
  • Estimated 1–5 API calls per client per sync depending on number of campaigns
  • Manual syncs triggered by agency staff only (not automated at high frequency)
  • Current client base: small number of managed accounts — well within Basic Access limits

We are requesting Basic Access (15,000 operations/day) which comfortably covers our current and projected usage.

7. Compliance & Data Handling

  • Campaign data is used solely to generate performance reports for the account owner (our client).
  • Data is never sold, shared with third parties, or used for any purpose other than reporting.
  • Each client can only view their own data — enforced via unique access tokens at the application layer.
  • We comply with the Google Ads API Terms of Service.
